# 提案資料作成スキル（戦略コンサルティング品質）

## 前提: analysis_context.md を必ず読み込んでから実行すること【最重要】

`data/docs/00_requirements.md` が存在する場合は必ず読み込む。
存在しない場合は `/data-analysis:data-requirements` を先に実行するよう案内する。

$ARGUMENTS にクライアント名・提案名・特別な指示があれば反映する。

## 提案資料の構成原則

### ピラミッドストラクチャー（バーバラ・ミント原則）
- **結論から始める**: 最初に「何を提案するか」を明示する
- **SCQ フレームワーク**: Situation（状況）→ Complication（問題）→ Question（問い）→ Answer（答え）
- **MECE 構造**: 論点が漏れ・ダブりなく整理されている
- **So What?**: すべてのファクトは「だから何か」に接続する

### 定量化の原則【GL-2, GL-4】
- すべての主張に数値根拠を付ける
- 推計値には前提条件と不確実性を明示する
- 感度分析で結論の頑健性を示す


### 2. `analysis_context.md` を更新

- セクション11（分析経過メモ）に「[日付]: 提案書作成完了」を追記

### 3. 提案書の品質チェック

**構造チェック**:
- [ ] エグゼクティブサマリーが1ページ（スライド1枚）相当に収まっているか
- [ ] SCQ フレームワークが明確に機能しているか
- [ ] すべての主張に定量的根拠が付いているか
- [ ] オプション比較がMECEか（漏れ・ダブりなし）
- [ ] 推奨案の根拠が明確か（他オプションを選ばない理由も含む）

**定量性チェック**【GL-1〜GL-5】:
- [ ] インパクト試算に感度分析があるか
- [ ] 推計値に95%信頼区間があるか【GL-4】
- [ ] 効果量（Cohen's d 等）が報告されているか【GL-2】
- [ ] 「相関≠因果」を誤用していないか【GL-9】

**誠実性チェック**【GL-7】:
- [ ] 仮説を否定する証拠も同等に提示しているか
- [ ] 不確実性・前提条件が隠されていないか
- [ ] チェリーピッキングが起きていないか

---

**不確実な点・仮定は必ず「不確実性: [内容]（確度: 高/中/低）」として明記すること。**
情報が不十分な場合は、仮定を明示した上で「追加情報があれば更新する」と注記して提案書を完成させる。
