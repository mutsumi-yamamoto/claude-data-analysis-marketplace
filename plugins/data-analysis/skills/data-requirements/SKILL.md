# 要件定義スキル（戦略コンサルティング品質）

## 前提: analysis_context.md を必ず読み込んでから実行すること【最重要】

$ARGUMENTS にミーティングメモ・コンテキスト文書が含まれている場合はそれを入力とする。
含まれていない場合は AskUserQuestion で「要件定義の元になるミーティングメモ・背景情報を貼り付けてください」と求める。

## Step 1: 入力の解析と情報抽出

$ARGUMENTS（またはユーザー提供テキスト）から以下を体系的に抽出する。
**不明・未言及の項目は空欄にせず「不確実性: [理由]（確度: 低）」と明記する。**

### 抽出チェックリスト

- [ ] **発注元（クライアント）**: 組織名・担当者・役職・意思決定権限
- [ ] **受注側（コンサルタント等）**: 組織名・担当者・役割
- [ ] **プロジェクトフェーズ**: 評価段階 / PoC / 本番構築 / 改善
- [ ] **ビジネス課題（As-Is）**: 現状の問題・痛みポイント・定量的損失
- [ ] **ユースケース候補**: 優先度付きリスト（明示的 + 暗示的）
- [ ] **利用可能データ**: ソース・形式・期間・件数・品質の所感
- [ ] **不足データ**: 必要だが現存しないデータ・追加取得の合意状況
- [ ] **ビジネスインパクト試算**: 提示された金額・%・工数等（根拠付き）
- [ ] **現行システム・ロジック**: 既存の仕組みとその限界
- [ ] **技術的制約**: 使用システム・技術スタック・インフラ
- [ ] **ビジネス制約**: 期限・予算・体制・組織的障壁
- [ ] **合意済みNext Step**: 次回打ち合わせ・追加提供物・宿題事項

## 壁打ちモード（要件を対話的に深める）

要件が曖昧な場合、または「壁打ちしながら要件を固めたい」と言われた場合は以下の問いを使って反復的に深掘りする。

**壁打ちの目的**: 「問題を正確に定義すること」。問題定義が間違っていれば正しい解決策を出しても無意味になる。

### 壁打ち質問セット（深さに応じて使い分ける）

**レベル1: 問題の本質を掘り下げる（最初に実施）**
- 「この問題が解決されると、誰にどんな良いことが起きますか？」（ゴールの明確化）
- 「現在この問題に対してすでに取り組んでいることはありますか？うまくいかなかった理由は？」（制約・失敗学習）
- 「もし今日から3ヶ月、この問題を無視したら何が起きますか？」（インパクトの具体化）

**レベル2: 優先度と制約を問う**
- 「提示された課題のうち、来週月曜日に1つだけしか解決できないとしたら何を選びますか？」（真の優先度）
- 「この分析結果を見て、『この数字なら○○する、でも○○なら別の判断をする』という分岐点はありますか？」（Actionabilityの確認）
- 「最悪のケースは何ですか？プロジェクトが失敗する最大のリスクは？」（リスク感度）

**レベル3: 仮説の反証を求める（GL-7）**
- 「『○○が原因』という仮説の反証証拠はありますか？この仮説が間違っている可能性は何%ですか？」
- 「今まで試したアプローチで『なぜ○○が機能しなかったか』の深い原因分析は行いましたか？」
- 「競合他社や業界のベンチマークと比較すると、この課題はどれほどユニークですか？」

**対話的な反復サイクル:**
```
提案 → 「この理解で正しいですか？」→ ユーザー確認 → 修正 → 再提案
```

各ラウンドで「前回から変わった点」と「まだ不明な点」を明示する。

## Step 2: ビジネス課題の構造化（Issue Tree）

Central Question を一文で定義し、MECE に分解する。

```
Central Question: [「何を解決すれば意思決定できるか」を一文で]

├── Issue A: [なぜ問題が起きているのか（原因診断）]
│   ├── Sub-Issue A-1: [仮説 / 根拠 / 確度]
│   └── Sub-Issue A-2: [仮説 / 根拠 / 確度]
│
├── Issue B: [どの程度の規模・インパクトか（インパクト定量化）]
│   ├── Sub-Issue B-1: [試算ロジック]
│   └── Sub-Issue B-2: [代替シナリオ]
│
└── Issue C: [どう解決できるか（解決策の評価）]
    ├── Option C-1: [実現可能性 / 期待効果 / 工数]
    └── Option C-2: [実現可能性 / 期待効果 / 工数]
```

各 Sub-Issue / Option に対して記載する:
| 項目 | 内容 |
|------|------|
| **仮説** | 「おそらく〇〇だろう」 |
| **根拠** | 入力文書から抽出した事実・数値 |
| **反証** | この仮説が誤りになる条件【GL-7】 |
| **確信度** | 高（根拠明確）/ 中（間接的根拠）/ 低（推測） |
| **検証方法** | 「○○データで○○を確認する」 |

**スコア解釈**:
- **20〜27点**: Phase 1（即時着手・PoC推奨）
- **10〜19点**: Phase 2（データ整備後・中期）

**優先度判定の注意点**:
- データ充足度が低(1)の場合、スコアに関わらず追加データ取得を前提条件とすること
- 複数の高優先度UCが競合する場合は、クリティカルパス観点で順序を決定すること

### 5-3. 最小検出水準（MDE）と必要サンプルサイズ【GL-5】

```python
from statsmodels.stats.power import TTestIndPower

analysis = TTestIndPower()
n = analysis.solve_power(
    effect_size=0.3,   # MDEをKGIから逆算して設定
    alpha=0.05,
    power=0.80,        # 検定力 1-β = 0.80
    alternative='two-sided'
)
print(f"必要サンプルサイズ: {n:.0f} / グループ")
```

現在のデータ件数と比較してギャップを記載。サンプル不足の場合は取得期間の延長・対象範囲の拡大を提案すること。
