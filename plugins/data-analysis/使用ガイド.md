## 1. プラグイン概要

このプラグインは、データ分析を **体系的・網羅的・恣意性なく** 実施するための Claude Code 拡張です。

### 特徴

| 特徴 | 内容 |
|------|------|
| **フレームワーク** | CRISP-DM拡張版（Phase 0〜8の9フェーズ） |
| **思考法** | McKinsey/BCGクラスの戦略コンサルタント行動規範 |
| **SSOT管理** | `analysis_context.md` を唯一の真実の情報源として全スキルが参照 |
| **バイアス防止** | GL-1〜GL-10の恣意的分析禁止ガイドラインを全スキルに適用 |
| **技術スタック** | Python 3（pandas, scikit-learn, scipy, statsmodels, xgboost, lightgbm, shap） |

## 3. セットアップ

### Step 1: テンプレートをプロジェクトルートにコピーする

```bash
# CLAUDE.md（自動読み込み設定）をコピー
cp plugins/data-analysis/templates/CLAUDE.md ./CLAUDE.md

# analysis_context.md（SSOT）をコピー
cp plugins/data-analysis/templates/analysis_context.md ./analysis_context.md
```

> **CLAUDE.md は Claude Code が会話開始時に自動で読み込みます。**
> これにより、毎回ルールを伝えなくても分析の前提条件が共有された状態で開始できます。

### Step 2: データファイルを配置する

```
your-project/
├── analysis_context.md   ← コピーしたSSOTファイル
├── data/
│   ├── your_data.csv     ← 分析対象データ
│   └── docs/             ← 分析レポートの出力先（自動作成）
└── .claude/
    └── commands/         ← スキルが参照するディレクトリ
```

### Step 3: 開始ウィザードを実行する

```
/data-analysis:da-start
```

## 4. 基本的な使い方

### パターンA: 標準フロー（推奨）

フェーズを順番に実行します。各フェーズ完了後に次へ進むため、途中で方針変更しやすいのが特徴です。

```
① /data-analysis:data-context    ← 必ず最初に実行
② /data-analysis:data-define
③ /data-analysis:data-collect    ← データがない場合
④ /data-analysis:data-integrate  ← 複数ソースを統合する場合
⑤ /data-analysis:data-explore
⑥ /data-analysis:data-clean
⑦ /data-analysis:data-feature
⑧ /data-analysis:data-model
⑨ /data-analysis:data-interpret
```

### パターンB: フルサイクル一括実行

Phase 0〜8 を一気に実行します。小規模プロジェクトや素早く結果を得たい場合に適しています。

```
/data-analysis:data-analyze
```

### パターンC: 途中フェーズから実行

`analysis_context.md` に進捗が記録されているため、特定のフェーズだけ単独で実行できます。

```bash
# 例: クレンジングだけやり直したい
/data-analysis:data-clean

# 例: レポートだけ作り直したい
/data-analysis:data-interpret
```

## 6. analysis_context.md の使い方

### 役割

**SSOT（Single Source of Truth）** ＝ 唯一の真実の情報源。

すべてのスキルは起動時にこのファイルを読み込みます。内容が不十分な場合は、分析に入る前にヒアリングを行います。

### 記載内容

| セクション | 内容 |
|-----------|------|
| プロジェクト基本情報 | 担当者・期限・ステータス |
| ビジネス背景・課題 | 分析が必要になった経緯・問題の内容 |
| 分析目的・Issue定義 | Central Question（解くべき問い） |
| 成功基準・KPI | KGI/KPI・定量的成功基準・MDE |
| スコープ | 対象期間・地域・データソース範囲 |
| データソース情報 | 各データの出所・件数・取得状況 |
| 仮説リスト | 仮説・根拠・検証手段・検証結果 |
| バイアスリスク評価 | 各バイアスのリスク評価と対策 |
| 成果物一覧 | 作成済みファイルと状態 |
| **分析経過メモ** | 分析を進める中で得た知見を時系列で記録 |

### 更新のタイミング

- スキル実行後、新たな知見が得られたら「分析経過メモ」に追記する
- 方針変更が生じた場合は理由とともに記録する
- 途中で再開する際は、このファイルを読むだけで文脈を復元できる

## 7. CLAUDE.md の役割

### CLAUDE.md とは

Claude Code が**会話開始時に自動で読み込む**設定ファイルです。プロジェクトのルールや構造を毎回伝えなくても、常にClaudeが把握した状態で分析を開始できます。

### このプラグインの CLAUDE.md 構成

| ファイル | 場所 | 用途 |
|---------|------|------|
| `CLAUDE.md`（マーケットプレイス用） | リポジトリルート | プラグイン開発のルールを定義 |
| `templates/CLAUDE.md`（分析プロジェクト用） | テンプレート | ユーザーが分析プロジェクトにコピーして使う |

### 分析プロジェクトへのセットアップ

```bash
# templates/CLAUDE.md をプロジェクトルートにコピー
cp plugins/data-analysis/templates/CLAUDE.md ./CLAUDE.md
```

コピー後のプロジェクト構造:

```
your-analysis-project/
├── CLAUDE.md              ← 自動読み込み（毎回の会話で有効）
├── analysis_context.md    ← SSOT（Claude が必ず最初に参照）
├── data/
│   ├── your_data.csv
│   └── docs/
```

### templates/CLAUDE.md に含まれる内容

Claude Code はこのファイルを読み込むことで、以下を自動的に把握します:

| 設定項目 | 内容 |
|---------|------|
| **最重要ルール** | `analysis_context.md` を必ず最初に読む |
| **ディレクトリ構造** | `data/`・`data/docs/` の役割 |
| **利用可能なスキル** | 10個のスラッシュコマンドと役割一覧 |
| **技術スタック** | Python 3・使用ライブラリ・エンコーディング規約 |
| **GL-1〜GL-10** | 恣意的分析禁止ガイドライン |
| **出力フォーマット** | ピラミッドストラクチャー・Next Step の形式 |
| **バイアスチェックリスト** | 全フェーズ共通の確認事項 |

## 8. 分析フローの全体図

```
[プロジェクト開始]
       ↓
analysis_context.md を配置
       ↓
/data-context ──── ヒアリング（一度に2〜3問）
       ↓             ↑ 回答が曖昧なら再ヒアリング
/data-define ───── Issue Tree / KGI/KPI / 仮説設計
       ↓
  データあり？
  ┌── Yes ──────────────────────────┐
  └── No → /data-collect            │
            → /data-integrate ──────┘
                                    ↓
                            /data-explore ── Phase 1: EDA
                                    ↓
                            /data-clean ──── Phase 2: クレンジング
                                    ↓
                            /data-feature ── Phase 3-5: 特徴量
                                    ↓
                            /data-model ──── Phase 6: モデリング
                                    ↓
                            /data-interpret ─ Phase 7-8: 解釈・報告
                                    ↓
                            [成果物を data/docs/ に出力]
```

## 8. 分析手法の選び方

スキル実行時に自動で選択されますが、目安として以下を参照してください。

```
目的変数がある？
├── Yes → 教師あり学習
│   ├── 連続値予測 → 回帰（線形回帰 → Ridge/Lasso → XGBoost/LightGBM）
│   ├── 二値分類  → ロジスティック回帰 → XGBoost/LightGBM
│   └── 多クラス  → 多クラス分類
└── No → 教師なし学習
    ├── グルーピング  → K-Means / DBSCAN
    ├── 次元削減      → PCA / t-SNE / UMAP
    └── 異常検知      → Isolation Forest / LOF

因果推論が必要？
├── 実験可能     → A/Bテスト（RCT）
└── 観察データ   → DID / RDD / 傾向スコアマッチング / 操作変数法
```

## 9. 品質ガイドライン（GL-1〜GL-10）

全スキルでこのガイドラインが適用されます。

| # | ルール | 目的 |
|---|--------|------|
| GL-1 | p値 < 0.05 のみ有意差。境界値は「限界的有意」と明記 | p値の過信防止 |
| GL-2 | 効果量を必ず報告。Cohen's d < 0.2 は「実務的に無意味」と明記 | 実務的有意性の確保 |
| GL-3 | 多重検定時はBonferroni/FDR補正を適用 | 多重比較問題の防止 |
| GL-4 | 点推定値には必ず95%信頼区間を付ける | 不確実性の可視化 |
| GL-5 | 分析前に検定力分析で必要サンプル数を明示 | サンプル不足の防止 |
| GL-6 | 外れ値除外はIQR×1.5 / 3σ等の客観的基準のみ | 恣意的除外の禁止 |
| GL-7 | 仮説を支持・否定する証拠を同等に提示 | 確証バイアスの防止 |
| GL-8 | グラフ軸は0始まりを原則。変更時は明記 | 視覚的誤解の防止 |
| GL-9 | 相関関係を因果関係として解釈しない | 相関≠因果の徹底 |
| GL-10 | 分析計画はデータ確認前に確定 | p-hackingの防止 |

## 10. 成果物一覧

スキル実行後、以下のファイルが `data/docs/` に自動生成されます。

| ファイル名 | 内容 | 対象読者 | 生成スキル |
|-----------|------|---------|----------|
| `01_eda_report.md` | EDAレポート | データサイエンティスト | `data-explore` |
| `02_cleaning_report.md` | クレンジングレポート | データサイエンティスト | `data-clean` |
| `05_feature_engineering.md` | 特徴量一覧・選定根拠 | データサイエンティスト | `data-feature` |
| `06_modeling_report.md` | モデリング・分析結果 | データサイエンティスト | `data-model` |
| `07_executive_summary.md` | エグゼクティブサマリー | 経営層・意思決定者 | `data-interpret` |
| `08_technical_report.md` | 技術的詳細レポート | データサイエンティスト | `data-interpret` |

レポートには必ず以下が含まれます:

- 分析目的とスコープ
- 使用手法と選択根拠
- 結果（数値・可視化）
- 限界・前提条件
- Actionableな示唆（Next Step: 誰が/何を/いつまでに）

## 11. よくある使い方パターン

### ケース1: 初めてデータ分析を行う

```
/data-analysis:data-context   # 目的・背景をヒアリング
/data-analysis:data-define    # KPI・仮説を設計
/data-analysis:data-collect   # 必要なデータを整理
/data-analysis:data-explore   # データの全体像を把握
（以降、順に実行）
```

### ケース2: データはあるが目的が曖昧

```
/data-analysis:data-context   # まずヒアリングで目的を明確化
/data-analysis:data-define    # Issue Treeで構造化
/data-analysis:data-explore   # データの実態を確認
```

### ケース3: 素早く分析結果を得たい

```
/data-analysis:data-analyze   # Phase 0〜8 を一括実行
```

### ケース4: 特定フェーズだけやり直したい

```
# 例: 特徴量を追加して再モデリング
/data-analysis:data-feature
/data-analysis:data-model
/data-analysis:data-interpret
```

### ケース5: 分析を中断して再開する

```
# analysis_context.md が進捗を保持しているため、
# 再開時も最初にコンテキストを読み込むだけでOK
/data-analysis:data-context   # 現状確認
（中断したフェーズから再実行）
```

## 12. 注意事項・禁止事項

### 必ず守ること

- **`analysis_context.md` を最初に配置する**: これがないと分析の文脈が共有されません
- **フェーズを飛ばさない**: 特にデータクレンジング（Phase 2）を省略すると、以降の分析結果が信頼できなくなります
- **ターゲットリーケージの確認**: 予測時に使えない情報が特徴量に含まれていないか、必ず確認します

### 禁止されていること（エージェントが自動チェックします）

| 禁止事項 | 理由 |
|---------|------|
| p値のみで有意差を判断する | 効果量・信頼区間との併用が必須（GL-2, GL-4） |
| 外れ値を感覚で除外する | 客観的基準（IQR法・3σ法）のみ使用可（GL-6） |
| 相関を因果として報告する | 因果推論は別途適切な手法が必要（GL-9） |
| 分析後に仮説を変更する | p-hackingに該当（GL-10） |
| 検定を何度も繰り返す | 多重比較問題が発生（GL-3） |
